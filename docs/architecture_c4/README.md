# Документация архитектуры C4 — LocalScript AI

## Уровень 1: Системный контекст

Система LocalScript AI представляет собой локальную платформу для генерации Lua-кода для платформы MWS Octapi LowCode. Основным пользователем системы является разработчик Lua-кода, который взаимодействует с системой через веб-интерфейс. Система работает полностью автономно в защищенном контуре заказчика. Единственным внешним взаимодействием является возможность выгрузки сгенерированного кода в репозиторий Git организации. Главной особенностью контекста является отсутствие любых соединений с внешними облачными AI-сервисами, что гарантирует безопасность данных и соответствие требованиям локализации.

## Уровень 2: Контейнеры

Архитектура системы состоит из нескольких ключевых контейнеров, взаимодействующих между собой внутри защищенной сети.
Клиентский слой реализован через Single Page Application на базе React и Vite. Это веб-интерфейс, который предоставляет пользователю возможности чата с агентом, загрузки документов для базы знаний и управления настройками генерации. Интерфейс общается с бэкендом исключительно по протоколу HTTPS внутри локальной сети.
Слой API реализован на фреймворке FastAPI. Он включает в себя несколько логических модулей. Модуль аутентификации отвечает за вход и регистрацию пользователей с использованием JWT токенов. Модуль генерации кода выступает оркестратором, управляющим процессом создания Lua-скриптов. Модуль RAG отвечает за поиск и индексацию документации в векторной базе данных. Модуль Sandbox обеспечивает безопасное выполнение кода в изолированной среде.
Слой данных включает в себя три основных компонента. PostgreSQL используется как реляционная база данных для хранения информации о пользователях, загруженных документах и истории генераций. ChromaDB выступает в роли векторной базы данных для хранения эмбеддингов документации, что необходимо для работы механизма RAG. Ollama служит средой выполнения для языковой модели, обеспечивая локальный инференс без внешних вызовов.

## Уровень 3: Компоненты

Внутри контейнера API Layer выделяются следующие ключевые компоненты.
Компонент агентского пайплайна управляет жизненным циклом запроса. Он принимает задачу от пользователя, при необходимости инициирует поиск контекста через RAG API, передает промпт языковой модели и получает сгенерированный код. Затем компонент отправляет код на валидацию в Sandbox API и возвращает результат пользователю.
Компонент работы с контекстом отвечает за формирование промпта. Он собирает вместе системные инструкции, историю диалога, переменные схемы wf.vars и найденные через векторный поиск фрагменты документации. Это позволяет модели генерировать код с учетом специфики платформы MWS Octapi.
Компонент валидации кода взаимодействует с Docker Sandbox. Он отправляет сгенерированный Lua-код на выполнение в изолированный контейнер с Lua 5.4. Это позволяет проверить синтаксическую корректность и отловить ошибки выполнения до того, как код будет показан пользователю.

## Уровень 4: Код и технологии

Реализация системы использует следующие технологии и подходы.
Фронтенд написан на TypeScript с использованием библиотеки React версии 18 и сборщика Vite. Клиентская часть включает сервисы для работы с API, управления аутентификацией и обработки потоковых ответов от сервера.
Бэкенд разработан на Python с использованием фреймворка FastAPI. Для организации агентского поведения применяется подход на основе графов состояний. Генерация кода осуществляется через локальный экземпляр Ollama с моделью Qwen2.5-Coder. Валидация происходит в Docker-контейнере с интерпретатором Lua 5.4.
Взаимодействие между компонентами происходит через внутренние HTTP запросы и SQL запросы к базам данных. Все данные шифруются и хранятся локально. Система поддерживает режимы работы как с использованием базы знаний RAG, так и в быстром режиме без 
поиска по документации.

## Безопасность и локальность

Архитектура спроектирована с приоритетом на безопасность данных. Все компоненты запускаются в изолированных Docker-контейнерах. Сетевое взаимодействие ограничено внутренней сетью Docker. Отсутствуют любые исходящие соединения к публичным API языковых моделей. Код, контекст схемы и бизнес-логика никогда не покидают инфраструктуру заказчика. Это обеспечивает полное соответствие требованиям защищенных контуров и позволяет развертывать систему на стандартном оборудовании без зависимости от внешних вендоров.
